# rag-platform-gitops

Kubernetes manifests and ArgoCD configuration for the multi-cloud RAG platform.

This repo is the GitOps source of truth for everything running on AKS. ArgoCD watches this repo and auto-syncs changes to the cluster.

## How it works

1. **Platform Helm apps** (`argocd/apps/*.yaml`) are applied manually via `kubectl apply`. They install Helm charts (NATS, KEDA, Crossplane, OTel Demo) and are managed by ArgoCD with `selfHeal: true`.

2. **Tenant manifests** (`tenants/*/`) are auto-discovered by the ArgoCD `ApplicationSet` (`root-appset.yaml`). Each subdirectory becomes an ArgoCD Application. Changes pushed to `main` are auto-synced to the cluster.

3. **Env vars and secrets**: Application deployments reference ConfigMaps and Secrets (`rag-ai-secrets`, `rag-backend-secrets`) that are created manually in the cluster (never in git).

## Key components

### NATS JetStream
Message broker for async ingestion. Stream `RAG` with subjects `rag.ingest` and `rag.ingest.repo`. The backend publishes, the worker consumes.

### KEDA
Autoscales the worker from 0 to 5 replicas based on NATS JetStream lag (threshold: 10 pending messages). No messages = no worker pods = no cost.

### Crossplane
Provisions Azure Cache for Redis (Basic C0) as a Kubernetes-native resource. The Redis instance serves as job status cache for the backend and worker.

### OTel Demo
OpenTelemetry Demo stack providing Grafana, Prometheus, OpenSearch, and Jaeger. These are the intended live observability data sources for the RCA agent hot path.

## How it fits in the platform

| Repo | Role | Tool |

| `rag-platform-infra` | Cloud infrastructure (AKS, VNet, Azure OpenAI, Firestore, WIF) | Terraform |
| `rag-platform-gitops` (this repo) | Kubernetes manifests, ArgoCD apps | ArgoCD + ApplicationSet |
| `rag-platform-app` | Application code + Docker images | GitHub Actions + GHCR |

## Conventions

- Platform Helm apps are applied manually (`kubectl apply`), not via ArgoCD self-management
- Tenant manifests are auto-synced by ArgoCD (`selfHeal: true`, `prune: true`)
- Secrets are created manually in the cluster, never committed to git
- Conventional commits for semantic-release (`feat:`, `fix:`, `chore:`)
